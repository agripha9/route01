## 0. 동기화 원칙 (먼저 읽기)

### 약관·개인정보처리방침은 **3곳**에 동시에 존재한다

변경 시 **세 곳을 모두 갱신해야 한다.** 한 곳만 빠뜨리면 사용자가 보는 내용 ≠ 가입 시 동의한 내용 ≠ 외부 등록(GCP) 내용이 되어 법적 분쟁의 빌미가 된다.

| 위치 | 파일·경로 | 노출 시점 |
|---|---|---|
| ① 정적 페이지 | `terms.html`, `privacy.html` | footer 링크, GCP 등록, 외부 공유, SEO |
| ② 회원가입 모달 | `nachim_v3.js` `R01_TERMS` 객체 | 온보딩 "보기" 버튼 (가입 동의 직전) |
| ③ 시행일·공고일 표기 | 위 두 곳 모두 | 모든 페이지 하단 |

**갱신 체크리스트**:
- [ ] `terms.html` / `privacy.html` 본문 수정
- [ ] `nachim_v3.js`의 `R01_TERMS` 객체 동일 내용으로 수정 (5448줄~)

## 2. 시행일·공고일 갱신 매뉴얼

### 약관 변경 시 절차

```
1. 변경 결정 → 변경 사항이 이용자에게 불리한가? 판정
   - 불리/중대 변경: 30일 사전 고지
   - 일반 변경: 7일 사전 고지

2. 공고일 = 오늘 날짜 (사용자에게 공지하는 날)
   시행일 = 공고일 + 7일 또는 30일

3. 다음 4곳 모두 갱신:
   a. terms.html / privacy.html 하단 "시행일: YYYY년 M월 D일"
   b. R01_TERMS 객체의 동일 부분
   c. (이용자 불리/중대 변경 시) 가입자 이메일로 변경 안내 발송 — Resend 사용
   d. 서비스 내 공지 (배너·모달 등 — 추후 공지 시스템 도입 시)

4. 시행일 도래 후 갱신된 약관 적용. 동의 거부자는 탈퇴 안내.
```

### 시행일 vs 공고일 위치
- 정적 페이지: `<p class="doc-meta">시행일: 2026년 5월 4일</p>`
- R01_TERMS: `<p style="margin-top:1.5rem;color:var(--ink3);font-size:12px">공고일: ... <br>시행일: ...</p>`

## 3. 갱신 작업 흐름 (실전 체크리스트)

### 체크리스트 — 매번 약관 변경 시

```
□ 1. 변경 사유·근거 문서화 (이 LEGAL_TRIGGERS.md의 어떤 트리거인지 명시)
□ 2. 영향받는 조항 식별 (위 표 참조)
□ 3. 이용자 불리 여부 판정 → 사전 고지 기간 결정 (7일 / 30일)
□ 4. 변경 초안 작성
□ 5. terms.html 또는 privacy.html 수정
□ 6. nachim_v3.js R01_TERMS 객체 동일 내용으로 수정
□ 7. 시행일·공고일 갱신 (정적 페이지 + R01_TERMS 양쪽)
□ 8. 캐시 버스터 v 번호 bump (CSS+JS)
□ 9. 코드 리뷰 (JS 문법, HTML 태그 균형, 캐시 버스터 일치)
□ 10. CONTEXT.md에 변경 내역 기록 (어떤 조항을 왜 변경했는지)
□ 11. 커밋 + 푸시
□ 12. 배포 확인 (route01.kr/terms, route01.kr/privacy)
□ 13. (이용자 불리 변경 시) 가입자 이메일 발송 — Resend
□ 14. (사용자 다수 영향 시) 서비스 내 공지
□ 15. 시행일 캘린더 등록 → 시행일 당일 별도 처리할 일 있는지 확인
```

## 5. 자주 잊는 함정 (메모)

- **R01_TERMS와 정적 페이지가 갈라지면 가장 큰 사고**. 한쪽만 갱신하고 잊는 경우 가장 흔함. 매번 양쪽 grep으로 확인.
- **시행일 = 변경 효력 발생일이지 게시일이 아니다**. "오늘 게시 = 오늘 시행"은 위법 (사전 고지 의무 위반).
- **위탁사 추가는 단순 추가가 아니라 §6(위탁) + §7(국외 이전) 동시 갱신**. 한국 사업자면 §7 면제, 해외 사업자면 §7 추가 필수.
- **분석 도구(GA·Mixpanel) 도입은 약관 4곳 갱신 (§2 자동수집, §6 위탁, §7 국외 이전, §11 쿠키)**. 한 곳만 갱신하기 쉬움.
- **AI 학습 데이터 사용 정책**은 사용자 신뢰의 핵심. 입력물을 모델 학습에 쓸지 여부, Anthropic 같은 위탁 LLM이 학습에 쓰는지 여부는 명확히 명시. (현재 Anthropic API는 기본적으로 학습에 사용하지 않음 — 약관 변경 시점에 Anthropic 정책 재확인 필요.)
- **가격 인상은 거의 항상 "이용자 불리"** → 30일 사전 고지 필수. 인하는 7일이면 충분.
- **사업자 등록 완료 시점**에 footer·약관 회사 정보, GCP 브랜딩, 토스페이먼츠 가맹점 정보 모두 갱신 필요. 이건 별도 마이그레이션 세션으로 묶어서 처리.
